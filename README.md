# 비동기 이미지 처리 시스템 과제

## 1. 기능 구현
- **이미지 처리 요청 API:** 클라이언트로부터 가공 요청을 수신하고 `taskId`를 즉시 반환합니다.
- **작업 상태 및 결과 조회 API:** 특정 작업의 진행 상태(PENDING, PROCESSING, COMPLETED, FAILED)와 결과를 조회할 수 있으며, 전체 목록은 `Pageable`을 활용해 페이징 처리하여 제공합니다.
- **외부 API 비동기 워커:** Kafka를 활용해 외부 `Mock Worker` 시스템과 HTTP 통신을 비동기로 수행하도록 구성했습니다.
- **장애 대응 및 재시도:** 외부 시스템의 응답 지연이나 실패에 대비해 지수 백오프(Exponential Backoff) 기반의 재시도 로직을 적용했습니다.

## 2. 주요 설계 및 기술적 고민

### 2.1 동시 요청과 멱등성 보장
동일한 요청이 중복으로 들어오는 경쟁 상태를 어떻게 방어할지 고민했습니다.
- 클라이언트가 헤더에 담아 보내는 `Idempotency-Key`를 활용해 식별자로 삼았습니다.
- 애플리케이션 레벨의 락(Lock)보다는 구조적인 방어가 확실하다고 판단하여, DB에 유니크 제약 조건(`uk_idempotency_key`)을 걸어 동시 요청을 차단했습니다. 중복 삽입 에러(`DataIntegrityViolationException`) 발생 시 기존에 생성된 `taskId`를 반환하도록 하여 일관성을 유지했습니다.

### 2.2 트랜잭션 경계 분리
외부 API 호출 시간이 수 초에서 수십 초까지 길어질 수 있다는 점이 가장 큰 문제였습니다.
- 이를 하나의 `@Transactional`로 묶을 경우 DB 커넥션 풀이 빠르게 고갈될 것이라 예상했습니다.
- 따라서 Worker 컴포넌트가 트랜잭션 밖에서 외부 API 통신을 전담하게 하고, Service 계층에서는 '상태 변경(PROCESSING)'과 '결과 마킹(COMPLETED/FAILED)'을 위한 아주 짧은 트랜잭션만 열리도록 분리했습니다.

### 2.3 외부 시스템 연동과 리소스 효율
- **가상 스레드(Virtual Threads) 도입:** I/O 블로킹이 길어지는 상황에서 기존 플랫폼 스레드를 사용하면 스레드 고갈이 발생할 수 있다고 판단했습니다. 이를 완화하기 위해 Java 21의 가상 스레드를 Kafka Consumer에 적용해 보았습니다.
- **명시적인 재시도 로직:** AOP 기반의 `@Retryable`은 내부 호출 제약이나 테스트 시 프록시 설정이 까다로워지는 단점이 있었습니다. 명확한 코드 흐름과 테스트 용이성을 위해 `RetryTemplate`을 빈으로 등록하고 명시적으로 주입받아 사용했습니다.

### 2.4 트래픽 증가 시 예상되는 병목과 개선 방향
현재 구조에서 트래픽이 몰릴 경우 아래와 같은 병목이 예상되며, 실무라면 다음과 같이 개선해보고 싶습니다.
1. **상태 조회를 위한 폴링 부하:** 클라이언트가 상태 확인을 위해 지속적으로 API를 호출하면 DB 읽기 부하가 심해집니다. 조회 빈도가 높은 상태 데이터는 Redis로 분산시키거나, SSE(Server-Sent Events)를 통해 서버가 완료 이벤트를 밀어주는(Push) 방식으로 개선이 필요할 것 같습니다.
2. **Kafka Consumer Lag:** 외부 API가 극단적으로 지연되면 큐에 메시지가 쌓이게 됩니다. 가상 스레드로 워커 서버의 스레드 고갈은 어느 정도 방어할 수 있겠지만, 본질적인 처리량을 높이려면 파티션 확장 및 워커 서버 스케일 아웃이 고려되어야 합니다.

## 3. 상태 모델과 장애 복구 처리

### 3.1 상태 전이 및 실패 처리
- **상태 모델:** `PENDING` ➔ `PROCESSING` ➔ `COMPLETED` / `FAILED`
- 도메인 엔티티(`ImageTask.java`) 내부에 상태 전이 검증 로직을 두어, 이미 처리 중인 작업이 PENDING으로 돌아가거나 완료된 작업이 다시 진행 상태로 꼬이는 것을 방지했습니다.
- Mock Worker와의 통신이 3회 재시도 후에도 실패하면 최종 `FAILED`로 마킹하고, 원인을 `result_data`에 기록하여 추후 추적이 가능하도록 했습니다.

### 3.2 메시지 처리 보장 (At-Least-Once)
- Kafka Consumer의 `enable-auto-commit`을 `false`로 설정했습니다.
- DB 상태 업데이트가 완전히 끝난 직후 `finally` 블록에서 수동으로 `ack.acknowledge()`를 호출하게 하여, 서버가 다운되더라도 메시지가 유실되지 않고 최소 한 번(At-Least-Once)은 처리되도록 구성했습니다.

### 3.3 서버 강제 종료 시의 한계점
- 처리 전(PENDING)인 메시지는 카프카를 통해 다시 소비되지만, DB 상태를 `PROCESSING`으로 변경한 직후 서버가 다운되는 경우에는 한계가 존재합니다.
- 카프카 메시지는 다시 들어오지만, 이미 도메인 로직에서는 '처리 중'인 상태로 인식해 예외를 던지며 처리를 거부하게 되고, 해당 작업은 영원히 `PROCESSING` 상태로 남게 됩니다.
- 이 문제를 온전히 해결하려면 일정 시간 이상 `PROCESSING`에 머물러 있는 태스크를 찾아 실패 처리하거나 다시 큐로 돌려보내는 별도의 작업이 필요합니다.

## 4. 제공 API 명세 요약
| HTTP Method | Endpoint | Description |
| :--- | :--- | :--- |
| `POST` | `/v1/images/tasks` | 헤더의 `Idempotency-Key`를 바탕으로 비동기 이미지 처리 작업을 요청합니다. |
| `GET` | `/v1/images/tasks/{taskId}` | 특정 작업의 현재 진행 상태 및 결과를 조회합니다. |
| `GET` | `/v1/images/tasks` | 전체 작업 목록을 페이징(`page`, `size`)하여 조회합니다. |

## 5. 테스트 전략
Mock에 과도하게 의존하기보다, 실제 인프라 환경에서의 동작을 보장하는 데 초점을 맞추었습니다.
- **통합 테스트 (Integration Test):** `Testcontainers`를 사용하여 실제 MySQL, Kafka 컨테이너를 일시적으로 구동한 상태에서 전체 흐름을 검증합니다.
- **E2E 검증:** `TestRestTemplate`을 사용하여 실제 HTTP 네트워크 통신 환경과 동일한 조건에서 API 응답 및 상태 코드를 확인합니다.
- **리포지토리 테스트:** 실제 DB 환경에서 유니크 제약 조건 등 스키마 제약 사항이 정상 작동하여 중복 저장을 방어하는지 검증합니다.

## 6. 로컬 환경 실행 방법

### 애플리케이션 빌드
```bash
./gradlew clean build -x test
```
### 인프라 및 앱 기동 (Docker Compose)
- docker-compose.yml을 통해 MySQL, Redis, Kafka 환경을 한 번에 구축할 수 있습니다.
```bash
docker-compose up -d --build
```
### 테스트 및 커버리지 확인
- JaCoCo 플러그인을 통해 테스트 성공 여부 및 커버리지 리포트를 생성할 수 있습니다.
```bash
./gradlew test jacocoTestReport
```
